**プロジェクト**: 柔術シミュレータ
**対象スコープ**: M1 — クローズドガード（攻防両方）限定
**前提文書**: [入力系詳細設計書 v1.0](./input_system_v1.md), [Visual Pillar Document v1.0](../visual/Visual_Pillar_Document_v1.docx)
**版**: 2026-04-18

## 0. 設計意図

入力系v1.0で参照されるが未定義の状態（グリップFSM、フックFSM、姿勢崩し値、判断窓、相手側の状態、スタミナ等）を **一つのドキュメント** に統一して定義する。決定の根拠:

- 入力系v1.0との相互参照を1ファイルに閉じる方が版管理が楽
- 状態間の依存（例: 姿勢崩し値が判断窓発火条件に直結）を一望できる
- 将来分割するとしても、まず統一版で全体像を確定させる方が分割線を引きやすい

### 0.1 共通の設計原則

1. **状態は所有者を持つ**: すべての状態は「攻め側」「防御側」「環境」のいずれかに所属する。判定時にどちらの入力で遷移するかが曖昧にならないようにする
2. **FSMは決定論的**: 同じ入力 + 同じ現在状態は常に同じ次状態を生む。AI/相手プレイヤーの不確実性は「入力」側に閉じ込める
3. **連続値とFSMを混在させない**: 連続的に変化する量（姿勢崩し値、スタミナ）はFSMの状態にせず、別途スカラ/ベクトルで持つ。FSMの状態数の爆発を避ける
4. **対称性原則（Q3 の決定）**: 攻め側と防御側は **同じ状態空間を持つ**。誰が「攻め」かは別レイヤー（§7 制御権レイヤー）で表現する

## 1. アクタとロール

### 1.1 アクタ

| アクタ | 説明 |
|---|---|
| `BottomActor` | クローズドガードを取っている側 |
| `TopActor` | パスを試みる側 |

両者は **同じ状態構造体（`ActorState`）** を持つ。クローズドガード成立中は `BottomActor` が攻め、`TopActor` が防御だが、これは制御権レイヤーで表現するので状態自体は対称。

### 1.2 ActorState の構成

```
ActorState {
    // 四肢の状態（§2）
    left_hand:  HandFSM
    right_hand: HandFSM
    left_foot:  FootFSM
    right_foot: FootFSM

    // 連続値（§3）
    posture_break: Vec2          // 姿勢崩しベクトル（β採用）
    stamina:       float [0, 1]  // スタミナ（§5）

    // 個別フラグ（§4）
    arm_extracted: { Left: bool, Right: bool }  // 腕が引き出された状態
}
```

`GameState` は次のように構成される:

```
GameState {
    bottom: ActorState
    top:    ActorState
    guard:  GuardFSM             // §6
    control: ControlLayer        // §7
    judgment_window: JudgmentWindowFSM  // §8
    time: TimeContext            // §9
}
```

## 2. 四肢のFSM

### 2.1 手のFSM（HandFSM）

入力系v1.0 §C.1.2 で参照されるグリップ探りステートマシンを正式定義する。

#### 2.1.1 状態と遷移

```
        ┌─────────────────────────────────────────┐
        ↓                                         │
     ┌──────┐  trigger_pressed   ┌──────────┐    │
     │ IDLE │ ────────────────→  │ REACHING │    │
     └──────┘                    └──────────┘    │
        ↑                              │          │
        │                       reach_complete    │
        │                              ↓          │
        │                        ┌─────────┐     │
        │                        │ CONTACT │     │
        │                        └─────────┘     │
        │                       ↙          ↘     │
        │             parry_check         grip_ok│
        │              fails               passes│
        │                ↓                   ↓   │
        │         ┌─────────┐         ┌─────────┐│
        │         │ PARRIED │         │ GRIPPED │┘
        │         └─────────┘         └─────────┘
        │              │                   │
        │       retract_done       trigger_released
        │              │             or grip_broken
        │              ↓                   ↓
        │         ┌─────────┐              │
        └──── RETRACT ←────────────────────┘
              └─────────┘
```

#### 2.1.2 状態定義

| 状態 | 意味 | 入場条件 | 出場条件 |
|---|---|---|---|
| `IDLE` | 手はニュートラル位置（自身の胸前） | RETRACT完了、初期化 | トリガ押下開始 → REACHING |
| `REACHING` | ターゲットへ移動中 | トリガ押下開始 | 移動完了（200–350ms、距離依存） → CONTACT |
| `CONTACT` | 1フレームのみの接触判定状態 | REACHING完了 | 同フレームで PARRIED か GRIPPED に分岐（§2.1.3） |
| `GRIPPED` | グリップ確立、IK剛性が trigger 値で変動 | CONTACT で grip 成功 | トリガ離放 / 相手のグリップカット成功 → RETRACT |
| `PARRIED` | 弾かれた瞬間（1フレーム） | CONTACT で parry | 即 RETRACT へ |
| `RETRACT` | ニュートラルへ戻り中（150ms） | PARRIED 経由 / GRIPPED 解放 | 完了 → IDLE。この間 REACHING 不可 |

#### 2.1.3 CONTACT での分岐判定

入力系v1.0 §C.2 を再掲・正式化する。判定順序は厳密に上から:

```
on CONTACT(this_hand, this_zone):
    if 相手の同zoneを手で防御中:                     → PARRIED
    elif 直近400ms以内に this_hand × this_zone で弾かれた: → PARRIED
    else:                                              → GRIPPED
```

「短期記憶」の400msは **同手・同ゾーンのみ** 適用。逆手で同ゾーンを狙うのは即座にOK。

#### 2.1.4 GRIPPED 中の追加遷移

GRIPPED中、次のいずれかで強制 RETRACT に入る:

| 引き金 | 出元 | 備考 |
|---|---|---|
| トリガ離放 | プレイヤー入力 | 通常 |
| `BTN_RELEASE` | プレイヤー入力 | 全グリップ即解放（§B.3） |
| 相手の「グリップカット」成功 | 相手側のアクション結果 | 詳細は§4.1 |
| 姿勢崩しによりターゲットが手の物理可達範囲外 | 状態自然変化 | 例: 相手が大きく仰け反ったため襟に手が届かない |

### 2.2 足のFSM（FootFSM）

入力系v1.0 §C.4 のフック状態を正式化する。

```
   ┌────────┐  bumper_edge   ┌──────────┐
   │ LOCKED │ ─────────────→ │ UNLOCKED │
   └────────┘                └──────────┘
        ↑                          │
        │                    bumper_edge
        │                          ↓
        │                    ┌──────────┐
        └─── lock_success ── │ LOCKING  │
                             └──────────┘
                                  │
                            lock_fail
                                  ↓
                            UNLOCKED へ戻る
```

| 状態 | 意味 |
|---|---|
| `LOCKED` | フックが相手の腰背面で組まれている。クローズドガード維持に寄与 |
| `UNLOCKED` | フックが解かれている。スイープや三角への遷移条件、または相手のパスが進む状態 |
| `LOCKING` | 再ロック試行中（300ms）。相手の姿勢が前傾（`posture_break.y ≥ 0.3`）なら成功、そうでなければ失敗で UNLOCKED に戻る |

両足同時に UNLOCKED → §6 ガード状態が CLOSED から OPEN へ遷移する。

## 3. 姿勢崩し値（PostureBreak）

### 3.1 モデル: 2次元ベクトル（Q2 採用）

`posture_break: Vec2` は **相手の上体が崩れている方向と量** を1つのベクトルで表す:

```
posture_break = (lateral, sagittal)
    lateral  ∈ [-1, 1]   // -1 = 自分の左へ崩れる, +1 = 右へ崩れる
    sagittal ∈ [-1, 1]   // -1 = 後傾（座った姿勢へ）, +1 = 前傾（顔が膝に近づく）
```

**採用根拠**:
- スカラ1次元では「前に崩れた」と「横に崩れた」を区別できず、技の発動条件を書き分けられない（例: ハサミは横崩し、ヒップバンプは前崩し）
- 6方向独立の冗長配列は組合せ爆発と整合性管理（合計1.0に制約等）が煩雑
- 2D ベクトルは「実装上スカラとほぼ同等のコスト」かつ「攻め側の意図する崩し方向と直接対応」する最小表現

### 3.2 大きさと「崩れている」判定

入力系v1.0 §D.1.1 で参照される「姿勢崩し値≥0.4」等の閾値は、**ベクトルの方向別射影** で判定する:

| 入力系v1.0 の表現 | 本文書での厳密化 |
|---|---|
| 「姿勢崩し値 ≥ 0.4」（無方向） | `‖posture_break‖ ≥ 0.4` |
| 「前方崩し ≥ 0.5」 | `posture_break.sagittal ≥ 0.5` |
| 「同側肩が前方崩し ≥ 0.6」 | `posture_break.sagittal ≥ 0.6 ∧ sign(posture_break.lateral) == 同側符号` |

技発動条件表（入力系v1.0 §D.1.1）の解釈は §8.2 で再掲する。

### 3.3 連続変化のルール

`posture_break` は次の合成で毎フレーム更新される:

```
posture_break ←
    posture_break * decay          (時定数 800ms で原点回帰)
  + bottom.hip_input * KHipImpact  (攻め側の腰の押し引き寄与)
  + grip_pulls * KGripImpact       (GRIPPED な手による引き寄せ寄与)
  - top.recovery_action            (防御側の戻し入力寄与)
```

各係数 `K*` のチューニングは M1 プレイテスト後に確定する。本文書は構造のみ規定する。

### 3.4 紙プロトとの対応

M0ペーパープロトタイプ（`m0_protocol_v1.md` §2.1）では姿勢崩し値を `0–4` の5段階に量子化している。本文書の連続2Dベクトルとの対応:

| ペーパー値 | 本文書対応 |
|---|---|
| `0` | `‖posture_break‖ < 0.1` |
| `1` | `0.1 ≤ ‖posture_break‖ < 0.3` |
| `2` | `0.3 ≤ ‖posture_break‖ < 0.5` |
| `3` | `0.5 ≤ ‖posture_break‖ < 0.75` |
| `4` | `‖posture_break‖ ≥ 0.75` |

**方向（lateral/sagittal）はペーパープロトでは欠落する**。これはM0が「条件成立の論理」を検証する設計（`m0_protocol_v1.md` §1.2）であり、技のベクトル方向区別までは検証スコープ外であることに整合する。

## 4. 個別フラグと派生状態

### 4.1 「腕が引き出された」状態（arm_extracted）

入力系v1.0 §D.1.1 三角絞めの条件で参照されるが未定義だった状態。

```
arm_extracted: { Left: bool, Right: bool }
```

**遷移条件**:

| 引き金 | 結果 |
|---|---|
| 相手側の `WRIST_*` または `SLEEVE_*` を `GRIPPED`(強度≥0.6) かつ攻め側 `hip_pull` 方向に1.5秒継続して引いた | 対応側を `true` |
| グリップが解放された、または相手が `BTN_BASE` ホールドで腕を引き戻した | 対応側を `false` |
| 5秒経過（自然リセット） | `false` |

「腕引き出し」は時間と継続を要する状態であり、瞬間的なグリップでは成立しない。ここに時間条件を入れることが「条件を作る」体験の核心。

### 4.2 グリップカット（相手側のアクション）

§2.1.4 で参照した「相手のグリップカット成功」を定義する。

防御側プレイヤー/AIが攻め側のグリップを切る試行は、自身の手で「攻め側の握っているターゲット部位」を払う動作:

| 条件 | 結果 |
|---|---|
| 防御側の手が IDLE/RETRACT、攻め側の手が GRIPPED な部位に対し「カット入力」 | 1.5秒の試行アニメーション開始 |
| 試行完了時、攻め側の `grip_strength < 0.5` | カット成功 → 攻め側は強制 RETRACT |
| 試行完了時、攻め側の `grip_strength ≥ 0.5` | カット失敗、防御側の手は IDLE に戻る |

防御側の入力詳細（カット入力のボタン割当）は **入力系v1.1（防御側）で確定** する。本文書は判定ロジックのみ規定する。

## 5. スタミナ（Stamina）

### 5.1 モデル

`stamina: float [0, 1]` の1次元スカラ。`1.0` が完全回復、`0.0` で「行動不能」。

### 5.2 消費と回復

| イベント | 影響 |
|---|---|
| 任意の手が `REACHING` または `GRIPPED`（強度0.5以上） | `-0.02/秒` |
| `posture_break` の絶対量が大きい状態を維持（攻め側） | `-0.05/秒 × ‖posture_break‖` |
| 判断窓中に技確定 | `-0.1`（一括） |
| `BTN_BREATH` 押下 + 全グリップなし + 静止（`hip_input < 0.2`） | `+0.1/秒` |
| 何もしていない（IDLE 全四肢、グリップなし） | `+0.03/秒` |

### 5.3 行動制約

- `stamina < 0.2`: グリップ強度の上限が `0.6` にクランプ。判断窓発火条件の `強度≥0.7` 系は不発になる
- `stamina < 0.05`: 新規 REACHING 不可。GRIPPED は維持できるが新たな試行は受け付けない

### 5.4 M1スコープでの扱い

M1ではスタミナの **消費・回復・閾値クランプのみ** 実装する。視覚化はVisual Pillar 2.4 のカラーグレーディング（呼吸の重さに連動した暖色シフト）で行い、HUD表示はしない。

## 6. ガード状態（GuardFSM）

```
   ┌────────┐  両足UNLOCKED   ┌──────┐
   │ CLOSED │ ─────────────→ │ OPEN │
   └────────┘                 └──────┘
                                 │
                            (M1スコープ外)
                                 ↓
                          [スクランブルへ]
```

**M1の挙動**: `CLOSED → OPEN` 遷移が発生した瞬間、本MVPはスクランブル領域に入るためセッションを **プレースホルダで終了**（即試合終了 or リセット）する。これは入力系v1.0 §C.4 と整合する。

`OPEN` から `CLOSED` への復帰はM1スコープ外。

## 7. 制御権レイヤー（ControlLayer）

Q3 で採用した「対称＋制御権レイヤー分離」の核心。誰が「今攻めている」かを **状態とは独立** に管理する。

### 7.1 構造

```
ControlLayer {
    initiative: enum { Bottom, Top, Neutral }
    initiative_locked_until: timestamp | None  // ロック中は変更不可
}
```

### 7.2 initiative の決定ルール

毎フレーム以下を評価して `initiative` を更新する。ロック中はスキップ:

| 条件（上から優先） | initiative |
|---|---|
| 判断窓が `OPEN` | 窓を発火させた側にロック（窓終了まで） |
| 任意のアクタが `arm_extracted == true` | 引き出した側 |
| 攻め側（クローズドガード保持側）の有効グリップ数 ≥ 2 | `Bottom` |
| 防御側のカット試行中 | `Top` |
| 上記いずれも不成立 | `Neutral` |

### 7.3 何のために使うか

`initiative` 自体はゲームロジックの状態遷移には影響しない。**カメラ・ポストプロセス・音響演出の判断材料** として使う:

- `initiative == Bottom`: カメラはやや前方寄り（攻め視点強調）、彩度+5%
- `initiative == Top`: カメラ後退、暖色シフト軽
- `initiative == Neutral`: ニュートラルグレード

これにより Visual Pillar の「HUDなしで状況を伝える」要件と、状態の対称性が両立する。

### 7.4 攻め/防御の入力解釈

入力系v1.0は「攻め側の入力」を主軸に書かれている（§G 未決事項）。本文書の対称構造により、防御側プレイヤーの入力は次のように扱える:

- `BottomActor` を操作するプレイヤー: 入力系v1.0 §A–§D の通り（攻め側）
- `TopActor` を操作するプレイヤー: §4.2 のカット試行、`BTN_BASE` のベース維持、`BTN_BREATH` を主に使う。詳細は **入力系v1.1（防御側） で確定**

## 8. 判断窓FSM（JudgmentWindowFSM）

入力系v1.0 §D.2 の判断窓ライフサイクルを正式FSM化する。

### 8.1 状態と遷移

```
   ┌────────┐ 条件成立 ┌─────────┐ 0.2s経過 ┌──────┐
   │ CLOSED │ ───────→ │ OPENING │ ───────→ │ OPEN │
   └────────┘          └─────────┘          └──────┘
       ↑                                       │
       │ 0.3s経過                               │  ↓ 技確定 / 見送り / 1.5s / 条件崩れ
       │                ┌─────────┐            │
       └────────────── │ CLOSING │ ←──────────┘
        cooldown 400ms └─────────┘
```

| 状態 | 持続時間 | 時間スケール |
|---|---|---|
| `CLOSED` | 不定 | 100% |
| `OPENING` | 0.2s | 100% → 30% に補間 |
| `OPEN` | 最大1.5s | 30% |
| `CLOSING` | 0.3s | 30% → 100% に補間 |
| `CLOSED`(クールダウン) | 0.4s（次の OPENING 不可） | 100% |

### 8.2 発火条件の参照（入力系v1.0 §D.1.1 の再掲・厳密化）

入力系v1.0 §D.1.1 の6技を §3 の `posture_break` ベクトル表現で再記述:

| 技 | 厳密条件 |
|---|---|
| ハサミスイープ | `bottom.left_foot==LOCKED ∧ bottom.right_foot==LOCKED ∧ (左右いずれかの SLEEVE_* が GRIPPED 強度≥0.6) ∧ ‖top.posture_break‖ ≥ 0.4` |
| フラワースイープ | `両足LOCKED ∧ WRIST_* GRIPPED ∧ top.posture_break.sagittal ≥ 0.5` |
| 三角絞め | `片足UNLOCKED ∧ top.arm_extracted の片側==true ∧ COLLAR_* GRIPPED` |
| オモプラッタ | `SLEEVE_* 片手GRIPPED ∧ top.posture_break.sagittal ≥ 0.6 ∧ sign一致 ∧ ｜bottom_hip_yaw｜ ≥ π/3` |
| ヒップバンプ | `top.posture_break.sagittal ≥ 0.7 ∧ bottom.hip_push ≥ 0.5 を 300ms維持` |
| 十字絞め | `両COLLAR GRIPPED 強度≥0.7 ∧ ‖top.posture_break‖ ≥ 0.5` |

### 8.3 候補集合

`OPENING` 入場時にスナップショットを取り、**OPEN中は候補集合を固定** する（OPEN中の条件追加成立は無視）。これにより「OPEN中に新たな技が点灯して混乱する」事故を防ぐ。

ただし **条件崩れによる強制 CLOSING**（§D.2 「相手が崩した場合」）は OPEN 中も継続監視する。候補集合のうち1つでも条件を維持していれば OPEN 継続、すべて崩れたら強制 CLOSING。

## 9. 時間コンテキスト（TimeContext）

判断窓中の時間スケールを管理する小さな状態:

```
TimeContext {
    scale: float    // 通常 1.0、判断窓中 0.3
    real_dt: float  // 物理フレーム間隔（実時間）
    game_dt: float  // = real_dt * scale、ゲームロジック用
}
```

**重要な原則**: 入力系A層のポーリング（§A.1）は **`real_dt` 基準で60Hz固定**。B層以降の意図変換とアニメーション補間は `game_dt` を使う。これにより判断窓中もプレイヤーは普通に入力でき、判断と操作の分離が成立する。

   │   ├─ stamina 連続更新            │

   └─────────────────────────────────┘
        ↓

各FSMは **同じフレーム内で1回ずつ遷移評価** され、上の順序で実行される。これにより「グリップ確立 → 同フレームで判断窓開く」が決定論的に処理される。

## 11. テスト戦略

### 11.1 ユニットテスト粒度

各FSMは独立にテスト可能:

- `HandFSM`: 入力イベント列 → 期待状態遷移列
- `FootFSM`: 同上
- `posture_break` 更新: 入力ベクトル列 → 期待ベクトル
- `JudgmentWindowFSM`: GameStateスナップショット → 発火可否

### 11.2 シナリオテスト

入力系v1.0 §D.1.1 の6技について、それぞれ「成立する入力列」「成立しない入力列（条件1つ欠ける）」を最小2ケースずつ書く。M1実装前にすべて緑であることを CI で担保する。

### 11.3 紙プロトとの整合検証

§3.4 の量子化対応に従い、M0で観測された「ペーパー上で判断窓が開いた局面」を本文書の連続値モデルで再現できるかを検算する。乖離があれば本文書 v1.1 で改訂。
