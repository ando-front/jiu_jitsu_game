# 入力系詳細設計書 v1.0

**プロジェクト**: 柔術シミュレータ
**対象スコープ**: M1 — クローズドガード（攻防両方）限定
**対象プラットフォーム**: Xbox コントローラ / DualSense（一級）、キーボード（動作確認用・機能制限あり）
**前提文書**: [Visual Pillar Document v1.0](../visual/Visual_Pillar_Document_v1.docx)
**版**: 2026-04-18

---

## 0. 設計意図

本文書は、プレイヤー入力がキャラクターの四肢駆動に変換されるまでの全パスを定義する。コア命題は次の通り:

> **「ボタン連打で技が出る」のではなく、「条件を作る」操作感を実現する。**
> 入力は技を発火させるトリガではなく、IKターゲットへの連続的な誘導であり、技は条件成立時に**自動的に開く判断窓**を経て発動する。

この命題を担保するため、入力系を**4層の独立したパイプライン**として設計する:

| 層 | 名前 | 役割 |
|---|---|---|
| A | 入力レイヤー | 物理デバイスのサンプリングと正規化 |
| B | 意図変換層 | 正規化入力→意味のある「意図」への翻訳 |
| C | アニメーション駆動層 | 意図→IKターゲット・グリップ試行への適用 |
| D | 判断窓トリガ層 | 状態の組み合わせを監視し、技発動の窓を開閉する |

各層は**単方向データフロー**で接続される（A→B→C→D）。Dから上位層へのフィードバックは、判断窓発動中のフラグ伝搬のみとする。

## A. 入力レイヤー

### A.1 サンプリング

- **ポーリング周期**: 物理フレーム同期（固定 60Hz、`16.67ms`）。可変フレームレート時もこの周期を厳守する。
- **入力遅延予算**: ボタン押下からアニメーションブレンド開始まで **≤2フレーム（33ms）**。これを超えると柔術家が「もたつき」と感じる閾値になる（一般格闘ゲームの許容値より厳しい設定。理由: 柔術はミリ秒単位のグリップ反応を伴うため）。
- **デバイス検出**: 起動時およびホットプラグ時に再列挙。最後にイベントを発したデバイスを「アクティブデバイス」とし、UIプロンプトの表示はこれに従う。

### A.2 ボタン・スティックマッピング

#### A.2.1 ゲームパッド共通レイアウト

| 物理入力 | 論理名 | 役割 | 備考 |
|---|---|---|---|
| 左スティック | `LS` | 腰・体幹の角度と押し引き | アナログ2軸 |
| 右スティック | `RS` | グリップ探り（手のターゲット先） | アナログ2軸 |
| L2 / LT | `L_TRIGGER` | 左手グリップ意図 (押下中) | アナログ0–1 |
| R2 / RT | `R_TRIGGER` | 右手グリップ意図 (押下中) | アナログ0–1 |
| L1 / LB | `L_BUMPER` | 左足アクション（フック解放/再ロック） | デジタル |
| R1 / RB | `R_BUMPER` | 右足アクション（フック解放/再ロック） | デジタル |
| ✕ / A | `BTN_BASE` | ベース取り（耐え／姿勢維持） | デジタル |
| ◯ / B | `BTN_RELEASE` | 全グリップ解放 | デジタル |
| △ / Y | `BTN_BREATH` | 呼吸を整える（スタミナ回復試行） | デジタル |
| □ / X | `BTN_RESERVED` | 予約（M1未使用） | — |
| Options / Menu | `BTN_PAUSE` | 一時停止 | — |

**設計根拠**:
- グリップを `L_TRIGGER`/`R_TRIGGER` に置いたのは、トリガのアナログ性が「握り込みの強度」と一致するため。0–1の値をそのまま握力として扱える。
- グリップの**ターゲット先**は `RS` で狙い、**握る/緩める**は `L_TRIGGER`/`R_TRIGGER` で表現する2軸構造を採る。これにより「どこを狙うか」と「どれだけ強く握るか」が独立する。Visual Pillar 4.2 のグリップファイト記述（袖を狙う→弾かれる→再度伸ばす）はこの分離があってはじめて表現できる。
- `BTN_BASE` は「耐える」で、相手の崩しに対するベース取りに割り当てる。クローズドガード攻め時は使用頻度が低いが、防御側操作で必須。

#### A.2.2 キーボードレイアウト（動作確認用・デジタル8方向）

Q1の決定に従い、**強度は常に最大値（=1.0）固定** とする。アナログ性を要するグリップ探りの「弾かれて戻る微振動」は再現されない旨を文書化する。

| キー | マップ先 | 値 |
|---|---|---|
| W / A / S / D | `LS` (8方向) | 軸ごとに ±1.0、斜めは正規化して `(±0.707, ±0.707)` |
| ↑ / ← / ↓ / → | `RS` (8方向) | 同上 |
| F | `L_TRIGGER` | 押下中 1.0、それ以外 0.0 |
| J | `R_TRIGGER` | 同上 |
| R | `L_BUMPER` | デジタル |
| U | `R_BUMPER` | デジタル |
| Space | `BTN_BASE` | デジタル |
| X | `BTN_RELEASE` | デジタル |
| C | `BTN_BREATH` | デジタル |
| Esc | `BTN_PAUSE` | — |

**制約事項（明示）**:
- グリップ強度のグラデーション、スティック傾きの強弱、入力の「揺らぎ」は再現できない。よってキーボード操作では**グリップファイトの探り合い演出は省略**され、`RS` のターゲット先指定が即座に最大強度で発火する。
- M1のプレイテストでは**ゲームパッドを必須デバイスとする**。キーボードは開発者の単体検証用と割り切る。

### A.3 デッドゾーンとカーブ

#### A.3.1 スティック

- **内側デッドゾーン**: 半径 `0.15`（正規化座標）。これ以下は `(0,0)` として扱う。
  - 根拠: DualSense / Xbox Eliteのドリフト実測上限を吸収する標準値。
- **外側デッドゾーン**: 半径 `0.95` 以上は `1.0` にクランプ。
  - 根拠: スティック端のメカニカルな引っかかりで「最大入力」が出ないケースを救済。
- **応答カーブ**: スティックは**指数カーブ `output = sign(x) * |x|^1.5`** を適用。
  - 根拠: 線形だと「ちょっと傾けただけで腰が大きく動く」感覚になる。柔術の腰の角度調整は中間域で精密性が要るため、中央付近を緩やかにする。

#### A.3.2 トリガ

- **デッドゾーン**: 下側 `0.05`、上側 `0.95`。
- **応答カーブ**: 線形のまま。トリガは「握る強さ」の物理メタファーがそのまま使えるため、加工しない。

### A.4 出力（B層への引き渡し）

A層は毎フレーム以下の構造体を1つ生成する:

```
InputFrame {
    timestamp: float (ms)
    ls: Vec2          // [-1, 1]^2、デッドゾーン・カーブ適用済み
    rs: Vec2          // 同上
    l_trigger: float  // [0, 1]
    r_trigger: float  // [0, 1]
    buttons: bitmask  // L_BUMPER, R_BUMPER, BTN_BASE, BTN_RELEASE, BTN_BREATH, BTN_PAUSE
    button_edges: bitmask  // 当該フレームで押下開始したボタンのみ立つ
    device_kind: enum { Xbox, DualSense, Keyboard }
}
```

`button_edges` を別途持つのは、B層で「押した瞬間」と「押しっぱなし」を区別する必要があるため（例: `BTN_BREATH` は押下エッジでのみスタミナ回復を開始する）。

## B. 意図変換層

A層の `InputFrame` を、ゲームロジックに意味のある「意図」へ翻訳する。

### B.1 LS → 腰の意図

`ls` を以下の3つの意図に分解する:

```
HipIntent {
    hip_angle_target: float (radian)  // 骸盤のヨー回転目標、対相手中心軸基準
    hip_push: float [-1, 1]           // +1=相手を押し離す、-1=相手を引き寄せる
    hip_lateral: float [-1, 1]        // ±1=左右への腰切り
}
```

**変換式**:
- `hip_angle_target = atan2(ls.x, ls.y) * KAngleScale`、ただし `KAngleScale = 0.6`
  - 根拠: スティックを完全に倒しても腰は ±π/2 までしか回らない。フル回転は別アクション（ヒップエスケープ）で行うため、通常入力では中庸範囲に抑える。
- `hip_push = ls.y` （前後成分そのまま）
- `hip_lateral = ls.x` （左右成分そのまま）

`hip_angle_target` と `hip_push` は冗長ではない: 例えば「正面を向いたまま押す」「45度切りながら押す」「45度切って引き寄せる」が独立に表現される。

### B.2 RS → グリップ意図

`rs` の方向ベクトルから、相手の身体を**ゾーン分割**したターゲット候補に最も近い1つを選ぶ。

#### B.2.1 グリップターゲットゾーン（クローズドガードの相手）

| ゾーン名 | スティック方向（攻め側視点） | 物理ターゲット |
|---|---|---|
| `SLEEVE_L` | 左下 | 相手の右袖口（プレイヤーから見て左前） |
| `SLEEVE_R` | 右下 | 相手の左袖口 |
| `COLLAR_L` | 左上 | 相手の右襟 |
| `COLLAR_R` | 右上 | 相手の左襟 |
| `WRIST_L` | 真左 | 相手の右手首（Gi越し / ノーギ） |
| `WRIST_R` | 真右 | 相手の左手首 |
| `BELT` | 真下 | 相手の帯 |
| `POSTURE_BREAK` | 真上 | 相手の後頭部（崩し用、両手必須） |

**ゾーン判定**:
- スティック方向の角度を8等分（45度刻み）し、最寄りゾーンを選択。
- ヒステリシス: 一度選んだゾーンから別ゾーンに切り替わるには、新ゾーン中心から ±15度 以内に入る必要がある。これがないと境界で手がブレる。

#### B.2.2 グリップ意図構造

```
GripIntent {
    l_hand_target: GripZone | None    // L_TRIGGER 押下中のみ非None
    l_grip_strength: float [0, 1]     // L_TRIGGER の値そのまま
    r_hand_target: GripZone | None    // R_TRIGGER 押下中のみ非None
    r_grip_strength: float [0, 1]
}
```

トリガを離した瞬間、対応する手のターゲットは `None` になり、C層で手が「ニュートラル位置」に戻る。

**重要な設計判断**: `RS` は「両手共通の狙い先」とする。L/R個別にしないのは、両手で同じターゲット（例: 両手で襟）を狙う頻度が高く、それぞれにスティックを割くと操作が破綻するため。L_TRIGGER と R_TRIGGER のどちらが押されているかで「今 RS が誰の狙いを決めているか」を切り替える。両方押されている場合は **両手とも同じターゲット**を狙う。

### B.3 ボタン → 離散意図

| ボタン | 意図 | エッジ/ホールド |
|---|---|---|
| `L_BUMPER` (押下エッジ) | 左足フック解放トグル | エッジ |
| `R_BUMPER` (押下エッジ) | 右足フック解放トグル | エッジ |
| `BTN_BASE` (ホールド) | ベース維持要求 | ホールド |
| `BTN_RELEASE` (押下エッジ) | 全グリップ即解放 | エッジ |
| `BTN_BREATH` (押下エッジ) | スタミナ回復試行開始 | エッジ |

`BTN_RELEASE` は「事故の出口」。プレイヤーが意図しないグリップを掴んでしまった時、即座に手放せる退避操作。これがないと操作の不可逆性が高まりすぎる。

### B.4 出力

B層は毎フレーム `Intent { hip: HipIntent, grip: GripIntent, discrete: [DiscreteIntent] }` を生成する。

## C. アニメーション駆動層

B層の意図を、Visual Pillar 4.2 の3層アニメーション構造（モーションキャプチャ ×プロシージャルIK ×物理ラグドール）に流し込む。

### C.1 IKターゲット計算

#### C.1.1 腰のIK

`HipIntent` から骨盤ボーンの目標トランスフォームを毎フレーム計算する:

- 目標ヨー回転: `hip_angle_target`
- 目標位置オフセット: `(hip_lateral * KLateralRange, 0, 0)`、`KLateralRange = 0.08m`
- 押し引き: `hip_push * KPushRange` の前後変位、`KPushRange = 0.05m`
- ブレンド速度: 現在値→目標値へ **時定数 `120ms`** の指数平滑。
  - 根拠: 即値追従だとカクつき、遅すぎると操作感が失われる。柔術の腰切りの体感速度に合わせた値。

#### C.1.2 手のIK（グリップ探り）

各手について、`GripIntent` のターゲットゾーンに対応する世界座標を **毎フレーム再計算** する（相手も動くため）。

**探り合いステートマシン**（各手ごとに独立）:

```
IDLE → REACHING → CONTACT → GRIPPED
                    ↓           ↓
                  PARRIED   (GRIPPED へ)
                    ↓
                  RETRACT → IDLE
```

- **IDLE**: トリガ離放中。手はニュートラル位置（自身の胸前）。
- **REACHING**: トリガ押下開始 〜 ターゲットへの移動中。所要時間 `200–350ms`（距離依存、線形）。
- **CONTACT**: ターゲットに手が触れた瞬間。1フレームのみ。次フレームで `GRIPPED` か `PARRIED` に分岐。
- **GRIPPED**: グリップ確立。`l_grip_strength` / `r_grip_strength` がIK拘束の剛性に直結する。
- **PARRIED**: 相手AIまたは相手プレイヤーの防御により弾かれた状態。
- **RETRACT**: 弾かれた手がニュートラルへ戻る。所要時間 `150ms`。この間、新しい `REACHING` 開始は **不可**（連打防止）。

**弾き判定**は次節 C.2 を参照。

### C.2 グリップ試行 vs 弾き判定

`CONTACT` フレームにおいて、相手の状態に応じて成否を判定する:

| 相手の状態 | 判定 |
|---|---|
| 同ターゲットゾーンを既に防御中 | `PARRIED` |
| プレイヤーの当該手が直近 `400ms` 以内に同ゾーンで弾かれた | `PARRIED`（短期記憶により再試行を弾く） |
| 上記以外 | `GRIPPED` |

「短期記憶」はAIの読み合いを成立させるための装置。これがないと「弾かれてもすぐ同じ場所を狙う」が最適戦略になり、グリップファイトが単調化する。

### C.3 トリガ強度→IK剛性

`GRIPPED` 状態では、`l_grip_strength` / `r_grip_strength` をIK拘束の剛性係数として使う:

- 強度 `0.0`: 手はターゲットに置かれているだけ。相手が動けば手は引きずられる。
- 強度 `1.0`: 完全拘束。相手の動きに対して手とターゲットが剛体接続。
- 中間値: 線形補間。

これにより**「軽く触れる」「強く握る」**の差がアニメーション上で自然に現れ、Visual Pillar 3.2 の「Giが伸びる量」「指の白さ」表現が連動可能になる。

### C.4 足フックのアクション

`L_BUMPER` / `R_BUMPER` のエッジ入力は、対応する足のフック状態をトグルする:

| 現状態 | エッジ後 |
|---|---|
| `LOCKED`（クローズド維持中） | `UNLOCKED`（足が離れる、ガードが開く方向） |
| `UNLOCKED` | `LOCKING`（再ロックを試みる、相手の姿勢次第で成否分岐） |

両足が同時に `UNLOCKED` になった場合、ガード状態は「クローズド → オープン」に遷移し、本MVPのスコープ外（スクランブル）に入る。**M1ではこの遷移をプレースホルダ（即試合終了 or リセット）として扱う**。

## D. 判断窓トリガ層

Visual Pillar 2.3 の判断窓を**自動で開く**条件を定義する（Q2の決定に従う）。

### D.1 判断窓の発火条件

判断窓は、特定の **状態の組み合わせ** が成立した瞬間に自動発火する。プレイヤーは入力を続けるが、ゲーム速度が30%に減速し、技発動ボタン（後述）が点灯する。

#### D.1.1 M1スコープの技と発動条件

| 技 | 条件（すべて同時に成立） | 技発動入力 |
|---|---|---|
| ハサミスイープ | 両足フック `LOCKED` & `SLEEVE_L`または`SLEEVE_R` を `GRIPPED`(強度≥0.6) & 相手の姿勢崩し値≥0.4 | `L_BUMPER` + `LS` 真横入力 |
| フラワースイープ | 両足フック `LOCKED` & `WRIST_L`/`WRIST_R` を `GRIPPED` & 相手の体重が前方≥0.5 | `R_BUMPER` + `LS` 上方向 |
| 三角絞め | 片足フック `UNLOCKED` & 相手の片腕が「引き出された」状態（別途状態管理）& `COLLAR_*` を `GRIPPED` | `BTN_BASE` 長押し（500ms） |
| オモプラッタ | `SLEEVE_*` を片手 `GRIPPED` & 相手の同側肩が前方崩し ≥0.6 & 自分の腰角度 ±60度以上回転 | `L_BUMPER` + `RS` 上方向 |
| ヒップバンプ | グリップなし可 & 相手の姿勢が前方崩し ≥0.7 & 自分の `hip_push ≥ 0.5` を `300ms` 維持 | `R_TRIGGER` 急速押下（200ms以内に0→1） |
| 十字絞め | 両手 `COLLAR_L` / `COLLAR_R` を `GRIPPED`(強度≥0.7) & 相手の姿勢崩し ≥0.5 | `L_TRIGGER` + `R_TRIGGER` 同時最大押下 |

**条件の意味**:
これらは「技を入力で出す」ためのコマンドではなく、**「条件が揃った後、最後のひと押しを入力で確定させる」** ための定義。プレイヤーは条件作りに集中し、判断窓が開いた瞬間に「どの技を選ぶか」を意思決定する。

### D.2 判断窓のライフサイクル

```
CLOSED → OPENING (0.2s, 速度100%→30%) → OPEN (max 1.5s) → CLOSING (0.3s) → CLOSED
                                            ↓
                                    プレイヤーが技発動入力 → 技確定
                                    プレイヤーが見送り(別入力) → CLOSING へ
                                    1.5s 経過 → CLOSING へ
```

- **OPEN中の入力受付**: D.1.1 表の「技発動入力」のみを技確定として受け付ける。それ以外の入力は通常通り B/C 層に流れるが、判断窓が閉じる引き金にはならない。
- **複数条件の同時成立**: 複数技の条件が同時に成立した場合、判断窓は開くが**点灯するボタンは複数**となる。プレイヤーが選んだ技が確定する。
- **判断窓中の中断**: 相手の入力により条件が崩れた場合（例: 相手がグリップを切った）、判断窓は **強制 CLOSING** に入る。技は不成立。

### D.3 連発抑制

判断窓が `CLOSED` に戻ってから次に開けるまで **クールダウン `400ms`** を設ける。条件のフラつきで連続発火することを防ぐ。

### D.4 出力

判断窓層は次のイベントを発行する:

- `JudgmentWindowOpened(candidate_techniques: [Technique])`
- `JudgmentWindowClosed(reason: enum { Confirmed(Technique), Cancelled, TimedOut, Disrupted })`

カメラ・ポストプロセス・音響・タイムスケールはすべてこのイベントを購読する。**入力系自身は演出を担当しない**（責務分離）。

## E. 全体データフロー

```
[Hardware]
   ↓ 60Hz polling
[A. Input Layer]
   → InputFrame
   ↓
[B. Intent Layer]
   → Intent (HipIntent + GripIntent + DiscreteIntent[])
   ↓
[C. Animation Driver]                 [D. Judgment Window]
   → IK targets, grip FSM transitions     ← reads game state
                                          → JudgmentWindowOpened/Closed events
   ↓                                     ↓
[Animation System] ←──────── time scale, camera blend triggers
[Camera / Post Process / Audio]
```

各層は**前段の出力構造体のみに依存**する。テスト時は任意の層に手書きの入力を流し込めるようにする（ユニットテスト容易性）。

## F. 検証指標

M1プレイテスト（20名）で本入力系を評価する基準:

| 指標 | 目標値 | 測定方法 |
|---|---|---|
| グリップファイト体感の再現性 | 15/20以上が「実際の探り合いと近い」 | プレイ後アンケート（5段階） |
| 入力レイテンシ体感 | 「もたつき」と感じた被験者 ≤2/20 | プレイ後インタビュー |
| 判断窓の自動発火が「不意打ち」でない | 「窓が開いた時、なぜ開いたか分かる」 ≥15/20 | プレイ後アンケート |
| 弾き判定の納得感 | 「弾かれた理由が分かる」 ≥15/20 | リプレイ視聴付きインタビュー |

これらの指標が未達の場合、本文書はv1.1へ改定する。
