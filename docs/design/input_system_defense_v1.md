## 0. 設計意図

入力系v1.0は攻め側（`BottomActor`）を主軸に書かれており、防御側操作は §G で未決事項として残されていた。本文書はそれを埋める。

### 0.1 設計上の縛り

1. **同じパッドで攻め/防御の両方が成立すること**: ボタン総数は変えない。各ボタンの **意味** が制御権（state_machines_v1.md §7）に応じて切り替わる
2. **対称性の維持**: 状態機械設計書 §1.2 で `ActorState` を対称に定義したため、防御側も同じ手のFSM・足のFSM・posture_break・stamina を持つ。本文書は **その入力ハンドラを定義** する
3. **防御側は「待つだけ」にしない**: 攻めの条件成立を妨害する3系統（カット / ベース / 戻し）と、能動的な前進1系統（パス試行）を持たせる
4. **HUDなし**: どちらを操作しているかは §6 のテキストプロンプト（起動時1回）+ カメラ位置で伝える

## A. 入力レイヤー（攻め側v1.0と共通）

サンプリング、デッドゾーン、応答カーブ、`InputFrame` 構造はすべて入力系v1.0 §A をそのまま流用する。**A層は攻め/防御を区別しない**。

区別は B層以降で「自分が `BottomActor` か `TopActor` か」を見て分岐する。

## B. 意図変換層（防御側固有）

### B.1 ロール判定

```
on every frame:
    my_role = (my_actor == GameState.bottom) ? Bottom : Top
    if my_role == Bottom:
        -> 入力系v1.0 §B の処理
    else:
        -> 本文書 §B.2 以降の処理
```

ロールは **セッション開始時に固定** される。M1ではセッション中に攻防が入れ替わる遷移（リガード等）はスコープ外。

### B.2 防御側のゲームパッドマッピング

ボタンの物理レイアウトは攻め側v1.0と同一。意味のみ変わる:

| 物理入力 | 論理名 | 攻め側の役割 | **防御側の役割** |
|---|---|---|---|
| 左スティック | `LS` | 腰・体幹の角度と押し引き | **トップポジションの体重移動** |
| 右スティック | `RS` | グリップ探り（手のターゲット先） | **カット対象 / ベース支点の方向指定** |
| L2 / LT | `L_TRIGGER` | 左手グリップ強度 | **左手のベース荷重**（押し付け強度） |
| R2 / RT | `R_TRIGGER` | 右手グリップ強度 | **右手のベース荷重** |
| L1 / LB | `L_BUMPER` | 左足フックトグル | **左手カット試行**（Q2: β採用） |
| R1 / RB | `R_BUMPER` | 右足フックトグル | **右手カット試行** |
| ✕ / A | `BTN_BASE` | ベース取り（使用頻度低） | **姿勢戻し**（崩されたposture_breakを能動的に戻す）|
| ◯ / B | `BTN_RELEASE` | 全グリップ解放 | **全ベース解放**（手を引く、パス試行への移行準備） |
| △ / Y | `BTN_BREATH` | 呼吸を整える | **呼吸を整える**（攻め側と同じ） |
| □ / X | `BTN_RESERVED` | 予約 | **パス試行コミット**（`RS`方向へパスを試みる） |
| Options | `BTN_PAUSE` | 一時停止 | 同左 |

#### B.2.1 各ボタンの設計根拠

- **`L_BUMPER`/`R_BUMPER` を「足フックトグル」から「カット試行」へ**: 防御側はクローズドガードに足を取られていないので、足のフックという概念自体が存在しない。バンパーの帯域が空くのでカットを割り当てる。Q2(β)の判断
- **`L_TRIGGER`/`R_TRIGGER` を「ベース荷重」へ**: 防御側にとって「手で押し付ける強さ」は攻め側の「握り込みの強さ」と物理メタファーが同型（連続的な力の入れ加減）。同じトリガを意味的に再利用できる
- **`BTN_BASE` を「姿勢戻し」へ**: 攻め側v1.0 では `BTN_BASE` は「耐え／姿勢維持」だった。防御側の「崩されたら戻す」は意味的にほぼ同じ（自分の安定を取り戻す行為）なので、ボタンの体感は変わらない
- **`BTN_RESERVED` (□/X) を「パス試行コミット」へ**: 攻め側で未使用だった枠を防御側固有の前進アクションに使う。コミット系（不可逆性が高い操作）はバンパー/トリガと混同しないよう独立ボタンが望ましい

### B.3 LS → トップポジションの体重移動意図

防御側の `LS` は腰の押し引きではなく、**重心の前後左右移動** を表す:

```
TopHipIntent {
    weight_forward: float [-1, 1]  // +1=相手の上に体重を乗せる, -1=後ろに引く
    weight_lateral: float [-1, 1]  // ±1=左右への重心スライド
}
```

**変換式**:
- `weight_forward = ls.y`
- `weight_lateral = ls.x`

`weight_forward > 0.5` を維持すると `top.posture_break.sagittal` が **減少方向** に作用する（自分の前傾を戻す。state_machines_v1.md §3.3 の `top.recovery_action` 寄与に該当）。

ただし `weight_forward > 0.5` は **同時にハサミスイープを成立させる条件にもなる**（攻め側から見て前崩し）。「攻めてくる相手に体重を乗せ返す」と「乗せすぎてスイープされる」のジレンマがここで生まれる。

### B.4 RS → カット対象 / ベース支点の方向指定

`RS` は文脈で2つの意味を切り替える:

| 状況 | RSの意味 |
|---|---|
| `L_BUMPER` または `R_BUMPER` を押下した瞬間 | 当該手のカット対象ゾーン（攻め側のグリップ位置）を指定 |
| バンパー押下なし、トリガ押下中 | 当該手のベース支点位置を指定（相手のどこに手を置くか） |
| どちらもなし | RSは無視される |

#### B.4.1 カット対象ゾーン

攻め側が握っている**自分の身体部位**の中から選ぶ。状態機械設計書 §2.1 に従い、攻め側の `HandFSM == GRIPPED` な手があれば、その握っているゾーン（`SLEEVE_*`/`COLLAR_*`/`WRIST_*`/`BELT`）が候補:

```
on L_BUMPER edge + RS:
    rs方向に最も近い、攻め側のGRIPPED中の自身の部位を選ぶ
    候補が0なら -> カット試行不発（音声フィードバックのみ）
    候補が1なら -> その手をカット
    候補が複数なら -> 最近候補を選択（ヒステリシスはなし、瞬間判定）
```

#### B.4.2 ベース支点ゾーン

トップ側が手を置ける相手側の部位（クローズドガード時）:

| ゾーン名 | RS方向 | 物理ターゲット |
|---|---|---|
| `CHEST` | 真上 | 相手の胸部（押し付けてポスチャ崩しに対抗） |
| `KNEE_L` | 左下 | 相手の左膝（パス試行の起点） |
| `KNEE_R` | 右下 | 相手の右膝 |
| `HIP` | 真下 | 相手の腰（押し下げて閉じたガードを解こうとする） |
| `BICEP_L` | 左上 | 相手の左二頭（コントロール用） |
| `BICEP_R` | 右上 | 相手の右二頭 |

ヒステリシス・8等分のロジックは攻め側v1.0 §B.2.1 と同一。

### B.5 トリガ → ベース荷重

`L_TRIGGER`/`R_TRIGGER` の値はそのまま「手の押し付け力」になる:

```
TopBaseIntent {
    l_hand_target: BaseZone | None       // L_TRIGGER 押下中のみ非None
    l_base_pressure: float [0, 1]        // L_TRIGGER の値そのまま
    r_hand_target: BaseZone | None
    r_base_pressure: float [0, 1]
}
```

**ベース荷重の効果**:
- `l_base_pressure ≥ 0.6` を `BICEP_*` に維持: 攻め側の同側 `arm_extracted` フラグが `false` 方向に押し戻される（state_machines_v1.md §4.1 の「相手が `BTN_BASE` ホールドで腕を引き戻した」に該当）
- `l_base_pressure ≥ 0.5` を `HIP` に維持: 攻め側の `posture_break.sagittal` が前傾方向に+される（攻めにとってはチャンス、防御にとってはリスク）
- スタミナ消費は攻め側のグリップ強度と同率（`-0.02/秒`、state_machines_v1.md §5.2）

### B.6 ボタン → 離散意図

| ボタン | 意図 | エッジ/ホールド |
|---|---|---|
| `L_BUMPER` (押下エッジ) | 左手カット試行（state_machines_v1.md §4.2）| エッジ |
| `R_BUMPER` (押下エッジ) | 右手カット試行 | エッジ |
| `BTN_BASE` (ホールド) | 姿勢戻し意図（自身のposture_breakを原点方向に追加減衰） | ホールド |
| `BTN_RELEASE` (押下エッジ) | 全ベース解放（両手を引く） | エッジ |
| `BTN_BREATH` (押下エッジ) | スタミナ回復試行開始（攻め側と同じ） | エッジ |
| `BTN_RESERVED` (押下エッジ) | パス試行コミット（§B.7） | エッジ |

#### B.6.1 姿勢戻し（`BTN_BASE` ホールド）の効果

ホールド中、毎フレーム以下が適用される:

```
top.posture_break ← top.posture_break * (1 - KRecover * dt)
    where KRecover = 1.5 / sec  (≈ 660ms の追加時定数)
```

通常の自然減衰（時定数 800ms）と合成され、ホールド中の実効時定数は約 **300ms** まで縮む。ただし以下の制約:

- `stamina < 0.3` のときは効果が半減（`KRecover = 0.75`）
- 攻め側のグリップ数が2本以上の場合も半減（強くコントロールされている状態では戻しにくい）

### B.7 パス試行コミット（`BTN_RESERVED`）

パス試行は「ガードを開けて通過する」ための **コミット入力**。M1スコープでは成功 = セッション終了（state_machines_v1.md §6 の `OPEN` 遷移と同等）。

#### B.7.1 コミット成立条件

`BTN_RESERVED` 押下エッジ時、以下をすべて満たすとパス試行が成立:

| 条件 | 理由 |
|---|---|
| 両手が `BICEP_*` または `KNEE_*` に荷重 ≥ 0.5 でベース | パス前提のコントロール |
| 攻め側の足フックが少なくとも片方 `UNLOCKED` | クローズドが完全に閉じている間はパス開始不可 |
| `RS` が下方向（`KNEE_*`/`HIP`）を指している | パスの方向意図 |
| 自身の `stamina ≥ 0.2` | 完全消耗ではコミットできない |

満たさない場合、押下は不発（演出のみ）。

#### B.7.2 コミット後の挙動

成立すると **5秒のパス試行アニメーション** に入る。この間:

- 攻め側はカウンター技1つを発動可能（M1ではプレースホルダ: 三角絞めのみ）
- カウンター成立 → パス失敗、攻め側勝利
- カウンター不成立 → パス成功、防御側勝利、セッション終了

**M1での簡略化**: パス試行アニメーションの詳細（中盤フェーズの細かい入力受付など）はスコープ外。コミット → 結果のみ実装する。

### B.8 出力

防御側B層は毎フレーム以下を生成する:

```
TopIntent {
    hip:      TopHipIntent      // §B.3
    base:     TopBaseIntent     // §B.5
    discrete: [DiscreteIntent]  // §B.6
}
```

C層（アニメーション駆動）は `BottomIntent` か `TopIntent` かをタグで分岐し、それぞれ別ハンドラに流す。

## C. アニメーション駆動層（防御側）

### C.1 体幹の制御

`TopHipIntent.weight_forward` / `weight_lateral` から、トップ側の骨盤+胸郭の合成重心を移動させる:

- 重心オフセット: `(weight_lateral * 0.10m, weight_forward * 0.12m, 0)`
- ブレンド時定数: `150ms`（攻め側の腰120msより少し遅い。トップポジションは支持点が多く、移動が物理的に重い体感を再現）

### C.2 手のFSM（ベース文脈）

防御側の手も状態機械設計書 §2.1 の `HandFSM` をそのまま使う。ただし状態の意味は文脈で変わる:

| HandFSM状態 | 攻め側での意味 | 防御側での意味 |
|---|---|---|
| `IDLE` | ニュートラル位置（胸前） | ニュートラル位置（自分の体側） |
| `REACHING` | グリップターゲットへ移動 | ベース支点へ移動 |
| `CONTACT` | 1フレーム接触判定 | 1フレーム接触判定 |
| `GRIPPED` | グリップ確立（IK剛性） | ベース確立（押し付け力） |
| `PARRIED` | 弾かれた | 該当しない（防御側のベース置きはほぼ常に成功する） |
| `RETRACT` | ニュートラルへ復帰 | 同左 |

**重要**: 防御側の `CONTACT` では基本的に `PARRIED` に入らない。攻め側との非対称はここに現れる（防御側のベース置きは攻め側のグリップより阻害されにくい。BJJのリアリティ反映）。

### C.3 カット試行のIK

`L_BUMPER`/`R_BUMPER` 押下エッジ時、当該手は現在の `HandFSM` 状態に関わらず **カット試行モーション（1.5秒）** に入る:

- 0–0.3秒: 当該手が `RS` で指定された攻め側のグリップ位置へ移動（IK）
- 0.3–1.2秒: 払う動作（手が外側へ振り抜かれる）
- 1.2–1.5秒: ニュートラル復帰

判定タイミングは `1.0秒` 時点。state_machines_v1.md §4.2 の判定ロジックに従う。

カット試行中は **当該手の `BaseIntent` を無視**（同時に2つのことはできない）。

### C.4 パス試行アニメーション

`BTN_RESERVED` コミット成立時、5秒の固定アニメーション再生。プレイヤー入力はM1では受け付けない（B.7.2 の通り）。

## D. 判断窓トリガ層（防御側固有の窓）

入力系v1.0 §D は攻め側の技発動条件のみを定義していた。本文書では **防御側の判断窓** も追加する。

### D.1 防御側の判断窓は「カウンターチャンス」

防御側にとっての判断窓は、**攻め側が技を仕掛けた瞬間** に開く:

```
on JudgmentWindowFSM transition: CLOSED -> OPENING (Bottom側発火):
    if その技に防御カウンターが定義されている:
        防御側にも同じ判断窓イベントを発行（候補=カウンター技）
```

### D.2 M1スコープのカウンター（最小実装）

| 攻め側の技 | 防御側カウンター | コミット入力 |
|---|---|---|
| ハサミスイープ | 体重の逆方向シフト | `LS` を攻めの sweep 方向と逆へ最大入力 |
| 三角絞め | 早期スタック（腕を抜く） | `BTN_BASE` 長押し + `LS` 真上 |
| その他4技 | M1ではカウンターなし（窓は開かない） | — |

カウンター成功時の効果:
- ハサミ逆: スイープ不成立、両者ニュートラルに戻る
- 三角早期スタック: 三角不成立、攻め側の `arm_extracted` が `false` に強制リセット

カウンターも **判断窓中に確定入力が必要**（自動防御ではない）。これにより防御側も「判断する側」として能動的にプレイできる。

### D.3 出力イベント

入力系v1.0 §D.4 と同じ `JudgmentWindowOpened` / `JudgmentWindowClosed` イベントを使う。識別子に「攻めの窓」「防御の窓」のタグを追加する:

```
JudgmentWindowOpened {
    side: enum { Bottom, Top }
    candidate_techniques: [Technique]
}
```

両側の窓は **同時に開いてもよい**（攻めが技を仕掛け、防御がカウンターチャンスを掴む同時局面）。時間スケール30%は1度だけ適用される（重複しない）。

## E. 攻め/防御の制御権切り替え（補足）

state_machines_v1.md §7 で定義した `ControlLayer.initiative` との関係:

| initiative | 入力解釈への影響 |
|---|---|
| `Bottom` | 演出のみ（カメラ寄り、彩度+5%）。入力ハンドラ自体は変わらない |
| `Top` | 同上（カメラ後退、暖色シフト軽） |
| `Neutral` | デフォルト演出 |

**入力ハンドラの分岐は `my_role`（プレイヤーが操作する側）で決まり、`initiative`（演出用）には依存しない**。これにより「演出が変わっても操作感は一貫」が保たれる。

## F. 起動時のロール提示（Q3: ii 採用）

セッション開始時、**1回だけ** 全画面テキストプロンプトを表示する:

```
あなたは [トップ / ボトム] です
[A] で開始
```

| 表示内容 | 持続 |
|---|---|
| ロール表示 | プレイヤーが `BTN_BASE` (✕/A) を押すまで |
| カメラはすでに該当ポジションに配置済み（OTS） | — |

セッション中は再表示しない。HUDなし原則を維持しつつ、初回の混乱を防ぐ最小介入。

## H. 検証指標（防御側、M1）

20名プレイテスト中、防御側を体験した被験者（推定半数）に対し:

| 指標 | 目標値 | 測定方法 |
|---|---|---|
| 「待つだけ」感の不在 | 「能動的に防いでいる感覚があった」 ≥ 8/10 | プレイ後アンケート |
| カット試行の納得感 | 「弾けた／弾けなかった理由が分かる」 ≥ 7/10 | リプレイ視聴付きインタビュー |
| パス試行コミットの慎重さ | 平均コミット回数 1–3 回/セッション | セッションログ |
| ロール認識の混乱 | 「最初に自分がどっちか分からなかった」と答えた被験者 ≤ 1/10 | プレイ後インタビュー |
